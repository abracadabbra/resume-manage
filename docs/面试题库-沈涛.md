## CH1. Java 基础 & 并发

### Q1: 线程池的核心参数有哪些？拒绝策略什么时候触发？

**考察点**：线程池配置是否真正理解，还是只会用。

**标准答案**：

| 参数 | 含义 |
|------|------|
| corePoolSize | 核心线程数 |
| maximumPoolSize | 最大线程数 |
| keepAliveTime | 空闲线程存活时间 |
| unit | 时间单位 |
| workQueue | 任务队列 |
| threadFactory | 线程工厂 |
| handler | 拒绝策略 |

**触发时机**：

```
提交任务 →
  1. 核心线程未满 → 创建核心线程执行
  2. 核心线程已满，队列未满 → 入队等待
  3. 队列已满，最大线程未满 → 创建临时线程执行
  4. 最大线程已满 → 执行拒绝策略
```

**四种拒绝策略**：
- `AbortPolicy`：抛RejectedExecutionException（默认）
- `CallerRunsPolicy`：由调用方线程执行
- `DiscardPolicy`：直接丢弃
- `DiscardOldestPolicy`：丢弃队列最旧的任务

**追问**：
- 你们风控系统的线程池怎么配置的？coreSize 和 maxSize 分别是多少？
- 为什么用 LinkedBlockingQueue 而不是 SynchronousQueue？
- 线程池的线程怎么保活的？keepAliveTime 设置多少？

### Q4: volatile 解决了什么问题？不能解决什么问题？

**考察点**：并发可见性 vs 原子性。

**volatile 能做到**：
- 可见性：写操作立即刷新到主内存，后续读操作直接从主内存读取
- 禁止指令重排序（内存屏障）

**volatile 不能做到**：
- 原子性：i++ 这种 read-modify-write 操作不是原子的

**经典追问**：
```java
public class Counter {
    private volatile int count = 0;

    public void increment() {
        count++;  // 非原子操作！
    }
}
```
- 这里 volatile 能保证线程安全吗？为什么？
- 正确答案：不能，需要 synchronized 或 AtomicInteger

### Q5: ConcurrentHashMap 的实现原理？JDK1.7 和 1.8 的区别？

**考察点**：并发容器的底层实现。

**JDK1.7**：
- Segment 分段锁，每个 Segment 内部是 HashEntry 数组
- 默认 16 个锁，减少锁竞争

**JDK1.8**：
- 去掉分段锁，改用 CAS + synchronized
- 锁的是头节点（bucket），粒度更细
- 数组+链表+红黑树（链表超过8转红黑树）

**追问**：
- 链表转红黑树为什么是8？不是7或9？
- size() 方法是怎么实现的？ConcurrentHashMap 怎么保证并发情况下计数准确？

### Q6: ThreadLocal 有什么用？有什么坑？

**考察点**：线程本地变量是否真正理解。

- 每个线程有自己的独立副本，线程间隔离

### Q12: 常见垃圾收集器？CMS 和 G1 的区别？

**考察点**：GC 调优能力。

| 收集器 | 算法 | 特点 | 适用场景 |
|--------|------|------|----------|
| Serial | 串行 | 单线程，STW | 单核、小数据量 |
| Parallel | 并行 | 多线程，关注吞吐量 | 后台批处理 |
| CMS | 并发 | 初始标记→并发标记→重新标记→并发清除，STW短 | 低延迟需求 |
| G1 | 区域化 | 分成多个Region，可预测停顿 | 大堆、多核 |

**CMS 的问题**：
- 浮动垃圾（并发清理阶段产生的新垃圾）
- 空间碎片（标记-清除）
- 耗尽CPU资源

**G1 的优势**：
- 可以设置 MaxGCPauseMillis
- 优先回收价值最大的Region
- 整合了标记-整理，不产生碎片

**追问**：

## CH4. 数据库 & 缓存

### Q25: MySQL 的索引结构？为什么用 B+ 树而不是 B 树/红黑树？

**考察点**：索引原理。

**B+ 树 vs B 树**：
- B+ 树只有叶子节点存储数据，非叶子节点只存索引
- B+ 树的叶子节点通过链表连接，范围查询友好
- B+ 树更扁平，磁盘IO次数更少

**B+ 树 vs 红黑树**：
- 红黑树高度不可控，深度可能很大
- B+ 树磁盘亲和性好，每个节点是一个磁盘页（16KB）

### Q27: MVCC 在 RR 级别下是怎么解决不可重复读的？

**考察点**：MVCC 深度理解。

**原理**：
- 每个事务有唯一的事务ID（递增）
- 每行数据有 `DB_TRX_ID`（创建版本）和 `DB_ROLL_PTR`（回滚指针）
- 读数据时：找创建版本<=当前事务ID，且删除版本不存在或>当前事务ID的数据
- RR 级别下，事务开始时生成 ReadView，后续都复用这个 ReadView

### Q31: Redis 的过期策略和淘汰策略？

**考察点**：内存管理。

**过期策略**：
- 定时删除：设置定时器，精确但CPU开销大
- 惰性删除：访问时检查，内存占用大
- 定期删除：每秒随机检查，平衡方案（Redis用的是这个）

**淘汰策略**：
- noeviction：不淘汰（默认）
- allkeys-lru / volatile-lru：LRU淘汰
- allkeys-random / volatile-random：随机淘汰
- allkeys-lfu / volatile-lfu：LFU淘汰
- volatile-ttl：淘汰TTL最短的

   - 连接池是否耗尽

5. **缓存**：
   - 是否缓存穿透
   - 是否大量miss

6. **外部依赖**：
   - 下游服务是否超时
   - 第三方API是否慢

**Java特定排查**：
```
jstack <pid>    // 看线程状态，死锁、阻塞
jmap -histo     // 看对象占用
jstat -gc       // 看GC频率
```
